# Programmatic Escrow Deployment

This doc describes how to deploy a **new escrow contract instance** each time a gig is successfully agreed between client and freelancer (when both `payee_acceptance` and `payer_acceptance` are true).

## Library: **viem**

Use **viem** for deployment. The app already uses viem for:

- Safe execution in `/api/execute-safe-tx` (wallet client, RPC, chain config)
- Wallet and chain setup in `WalletProvider` and `lib/magic.ts`

Using viem keeps one consistent stack and avoids mixing ethers and viem for the same chain. viem’s `walletClient.deployContract()` is the right primitive.

## Who deploys and who pays gas

One option:

| Approach | Who deploys | Who pays gas | Use case |
|----------|-------------|--------------|----------|
| **Backend relayer** | Your API (with `RELAYER_PRIVATE_KEY`) | Server relayer (same as Safe execution) | One escrow per agreed gig; platform sponsors gas. |


Recommended for “deploy when gig is agreed”: **backend relayer**. Same gas model as `execute-safe-tx`, and you can store the new `escrowAddress` in `contractSpecs` right after deployment.

## Deployment flow

1. **Trigger**  
   When a contract moves to “agreed” (e.g. both acceptances true), your backend or frontend calls the deploy endpoint (e.g. `POST /api/deploy-escrow`).

2. **Backend**  
   - Load escrow ABI and bytecode (from env, build output, or a shared module).
   - Build constructor args (e.g. client, freelancer, amount, token, dispute timeout).
   - Create a viem `WalletClient` with `RELAYER_PRIVATE_KEY` and the same chain (e.g. Polygon) you use in `execute-safe-tx`.
   - Call `walletClient.deployContract({ abi, bytecode, args })` → get `txHash`.
   - Optionally `publicClient.waitForTransactionReceipt({ hash: txHash })` and read the deployed address from the receipt (or derive it with `getContractAddress` from deployer + nonce).
   - Return the deployed escrow address (and optionally `txHash`).

3. **Persist**  
   Save the new address in the contract’s `contractSpecs`, e.g.:

   ```ts
   contractSpecs: [{ network: "polygon", escrowAddress: deployedAddress }]
   ```

4. **Later**  
   Funding, release, and dispute use this `escrowAddress` (e.g. via Safe or direct calls), using the same relayer pattern if you want gas sponsorship.

## Bytecode and ABI

You need the escrow contract’s **compiled bytecode** and **ABI**. Prefer loading them from build output so they stay in sync with the contract.

### Where things should live

| What | Ideal location | Notes |
|------|-----------------|--------|
| **Solidity source** | `contracts/` at repo root (e.g. `contracts/Escrow.sol`) | Single source of truth; compile with Foundry or Hardhat. |
| **Build output (bytecode + ABI)** | `contracts/out/Escrow.sol/Escrow.json` (Foundry) or `artifacts/contracts/Escrow.sol/Escrow.json` (Hardhat) | Generated by `forge build` or `npx hardhat compile`. Do not commit if you prefer; then the API can read from a build step that runs before deploy. |
| **Consumption at the API** | **Next to the API or in `lib/`** — e.g. `lib/escrow-artifact.ts` or `app/api/deploy-escrow/escrow-artifact.ts` | One module that imports the JSON artifact (or reads it at runtime) and exports `escrowAbi` and `escrowBytecode`. The deploy route then imports from this module instead of holding ABI/bytecode inline or in env. |
| **Secrets** | `.env` (server-only) | `RELAYER_PRIVATE_KEY` only. Do not put bytecode in env; it’s large and will drift from the contract. |
| **Request payload (constructor args)** | Request body to `POST /api/deploy-escrow` | e.g. `payer`, `payee`, `amountWei`, `token`. These are per-deployment variables, not build artifacts. |

### Recommended flow at the API

1. **Add a small artifact module** (e.g. `lib/escrow-artifact.ts`) that pulls from the build output:
   - **Foundry**: `import escrowArtifact from '@/contracts/out/Escrow.sol/Escrow.json'` (or `readFileSync` if the path is outside the bundle), then export `escrowArtifact.abi` and `escrowArtifact.bytecode.object`.
   - **Hardhat**: Same idea with `artifacts/contracts/Escrow.sol/Escrow.json` (path may differ; adjust for your Hardhat config).
2. **In `app/api/deploy-escrow/route.ts`**: Import `escrowAbi` and `escrowBytecode` from that module. Read constructor args from the request body. Use env only for `RELAYER_PRIVATE_KEY` and optionally `NEXT_PUBLIC_POLYGON_RPC_URL`.
3. **Build pipeline**: Run `forge build` (or Hardhat compile) before `next build` so the artifact exists when the API runs. You can add a `prebuild` script or CI step.

This keeps the deploy route thin, avoids huge env vars, and ensures the API always uses the same bytecode/ABI as the compiled contract.

## Summary

- **Library**: viem (`deployContract` on `WalletClient`).
- **Who deploys**: Backend with `RELAYER_PRIVATE_KEY` (recommended).
- **Process**: Trigger on gig agreed → API deploys with viem → return escrow address → store in `contractSpecs`.
- **Gas**: Paid by the same relayer as Safe execution; no change for the user.
